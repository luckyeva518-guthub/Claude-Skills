# CDQ v3 报告输出规范

> 本文件定义 CDQ v3 诊断报告的输出格式规则，所有报告必须遵守。

---

## 一、报告格式

### 1.1 输出文件
- 每次诊断同时输出 **MD + HTML** 两种格式
- 文件命名：`CDQ_v3_Report_{ASIN}.md` 和 `CDQ_v3_Report_{ASIN}.html`
- 保存路径：用户桌面（`C:\Users\{username}\Desktop\`）

### 1.2 版本号
- **报告中不出现任何版本号**（如 v1.0、v1.3.0 等）
- 版本信息仅记录在 SKILL.md 的变更日志中

### 1.3 报告结构（按顺序）
1. 基本信息
2. 总分 + 等级
3. 各维度得分汇总表
4. 政策违规检测（U 级风险扫描）
5. 维度 1：结构化属性质量
6. 维度 2：标题质量
7. 维度 3：变体质量
8. 维度 4：产品类型分类
9. 维度 5：图片质量
10. 维度 6：五点描述
11. 维度 7：A+ 页面
12. 高频词检查
13. 整体合规性
14. 缺陷清单
15. 优化优先级建议
16. 预期优化效果（分阶段）
17. 附录：商业表现参考（不直接影响评分）

---

## 二、数据来源规则

### 2.1 前台优先
- **五点描述 (BP) 数量以前台抓取数据为准**，后台 API 仅返回部分 BP 是已知 bug
- 前台已验证存在的内容，状态标记为 ✅，不因后台数据不全而标 ⚠️
- 后台 API 数据不全不作为 Listing 缺陷扣分

### 2.2 数据差异标注
- 前后台数据不一致时，在报告中注明差异（如评论数、标题内容）
- 不重复标注前台已验证通过的问题

### 2.3 Search Terms 字段
- Catalog API 不返回 Search Terms 字段，工具不支持查询
- 报告中标记为 ⚠️（非 ❌），说明"工具不支持查询，请至卖家后台确认是否已填写完整"
- 缺陷清单中优先级为 🟡 中等（非 🔴 严重），因为无法确认实际状态

---

## 三、状态标记规则

### 3.1 ✅ / ⚠️ / ❌ 使用标准
| 标记 | 含义 | 示例 |
|------|------|------|
| ✅ | 合格/已填写/已验证 | BP 前台验证 5 条 ✅ |
| ⚠️ | 需关注/不完整/有风险 | Material 仅填写部分 ⚠️ |
| ❌ | 缺失/错误/未填写 | Search Terms 字段不存在 ❌ |

### 3.2 优先级标记
- 🔴 严重：影响评分 ≥2 分或存在政策风险
- 🟡 中等：影响评分 1~2 分
- 🟢 低：加分项缺失或轻微问题

---

## 四、不输出内容

以下内容 **不在报告中呈现**：

- ~~数据采集工具状态~~（MCP 工具是否可用、web_reader / analyze_image 状态）
- ~~版本号~~（v1.x.0 等）
- ~~GPSR 合规评估~~（仅作为数据完整性问题评估，不作为合规风险评分，footer 中注明"GPSR 合规评估已排除"即可）

---

## 五、HTML 报告样式

### 5.1 整体风格
- 单页 HTML，内联 CSS，不依赖外部资源
- 响应式布局，移动端适配
- 配色：蓝白为主，绿/橙/红用于状态标记

### 5.2 核心组件
- **Header**：深蓝渐变背景，报告标题 + 诊断时间 + 数据来源
- **Score Card**：居中大字显示总分和等级
- **Card**：每个维度/章节一个卡片，圆角 + 轻阴影
- **表格**：蓝灰表头，hover 高亮行
- **Alert**：红/橙/绿三种级别的提示框
- **Footer**：数据来源说明

### 5.3 参考实现
以 `CDQ_v3_Report_B0BWY7ZDT2.html` 为标准模板，后续报告保持一致的 CSS 变量和组件结构。

---

## 六、报告尾部固定内容

每份报告 Footer 统一为：
```
本报告由 CDQ v3 诊断 Skill 生成
数据来源：后台 Catalog API + 前台 web_reader 抓取 + 图片 analyze_image 分析 | GPSR 合规评估已排除
```
